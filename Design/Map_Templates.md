# 1. 设计原则

## 1.1 房间是节拍序列，不是关卡

每个房间是 **20-40 秒可走完**的节拍单元。一个 Run = 3-8 个房间拼接。

| 单位 | 时长 | 节拍数 |
|------|------|--------|
| 单房间 | 20-40 秒 | 5-10 个信息事件 |
| 单区域（3 房间） | 60-120 秒 | 15-30 个 |
| 完整 Run（5-7 房间） | 5-12 分钟 | 35-70 个 |

## 1.2 节拍密度硬指标

按 [Sensory_Design §6](Sensory_Design.md#6-信息密度要求) 的强度分级，**每个房间至少包含**：

- **2 个"极强"节拍**（犹豫触发点、性格高亮触发点）
- **3 个"强"节拍**（感知触发、状态切换、战斗）
- **若干"中"节拍**（揭开新区域、朝向变化）

**做不到这个密度 = 房间不合格，删掉重做**。

## 1.3 房间不能"空"

任何房间不能有**连续 8 秒**没有"中/强/极强"事件。设计时必须沿生物可能走的路径**逐秒走一遍**，标记每秒发生什么。空白超 5 秒就加节拍元素。

# 2. 节拍元素清单

每个房间从以下元素库中**至少选 5 种**：

| 元素 ID | 名称 | 触发反应 | 玩家学到 | 强度 |
|---------|------|----------|---------|------|
| **B-Fork** | 视野分叉 | 减速 + 扫描 | 性格如何决断 | 强 |
| **B-Smell-Src** | 气味源 | 嗅觉提前感知 | 嗅觉早于视觉 | 强 |
| **B-Corner** | 视觉阻挡转角 | 减速 + 朝向锁定 | 信息层差 | 中 |
| **B-Bait** | 诱饵（看似资源实为陷阱） | 贪欲生物上当 | 性格的代价 | 极强 |
| **B-Trail** | 痕迹（脚印/血迹/抓痕） | 短期记忆触发 | 环境叙事 | 中 |
| **B-Resource** | 资源点 | 拾取行为 | 优先级取舍 | 强 |
| **B-Hazard** | 危险地形（酸池/毒气） | 抗性决定通过 | 身体决定能力 | 强 |
| **B-Narrow** | 狭窄通道 | 体型约束（Post-MVP） | 身体的约束 | 中 |
| **B-Enemy-Patrol** | 巡逻敌人 | 警戒 + 战斗或绕行 | 性格主导战斗 | 极强 |
| **B-Enemy-Ambush** | 伏击敌人（藏在视觉阻挡后） | 突然出现 + 情绪层 `!` | 信息层差的代价 | 极强 |
| **B-Chest** | 宝箱 | 拾取 + 短犹豫 | 奖励节拍 | 强 |
| **B-Memory-Cue** | 记忆触发点（曾经死过同伴的位置等） | 朝向锁定 + 减速 | 历史敏感 | 强 |
| **B-Smell-Maze** | 气味迷宫（多个气味源冲突方向） | 嗅觉混乱 + 犹豫 | 信息不完美 | 极强 |
| **B-Dark** | 黑暗区（视觉范围缩小） | 减速 + 朝向扫动 | 感知模态的局限 | 中 |
| **B-Exit** | 出口 | 任务完成或撤退 | 终点 | 强 |

**MVP 阶段必须实现**：B-Fork / B-Smell-Src / B-Corner / B-Resource / B-Chest / B-Exit / B-Enemy-Patrol / B-Trail / B-Memory-Cue。

**Post-MVP**：B-Bait / B-Hazard / B-Narrow / B-Enemy-Ambush / B-Smell-Maze / B-Dark。

# 3. 房间模板定义结构

```csharp
public sealed class RoomTemplate
{
    public string TemplateId;              // R01-Lab / R02-FoyerSmell / ...
    public RoomCategory Category;          // Explore / Encounter / Resource / Transition / Anomaly
    public RoomDifficulty Difficulty;      // T1 / T2 / T3
    public Vec2Int Size;                   // 单位 m，典型 20×20
    public List<ConnectionPort> Ports;     // 出入口
    public List<BeatSlot> BeatSlots;       // 节拍元素插槽（位置 + 元素类型 + 随机变体）
    public List<StaticObstacle> Obstacles; // 墙、矮丛、岩石
    public PathHint[] PrimaryPaths;        // 设计师预想的主要路径（用于密度自检）
}

public sealed class BeatSlot
{
    public Vec2 Position;
    public string BeatElementId;           // B-Fork / B-Smell-Src / ...
    public string[] Variants;              // 该插槽允许的随机变体
    public bool IsRequired;                // 必出，还是可选
}
```

## 3.1 房间类别

| 类别 | 节拍主调 | 出现频率 | 难度上限 |
|------|----------|----------|---------|
| **Explore** 探索室 | 无敌人，多分叉/痕迹/气味 | 40% | T1-T3 |
| **Encounter** 遭遇室 | 1-2 敌人，地形复杂 | 30% | T1-T3 |
| **Resource** 资源室 | 高价值资源 + 守卫/危险 | 15% | T2-T3 |
| **Transition** 转场室 | 窄通道/黑暗，紧张积累 | 10% | T1-T3 |
| **Anomaly** 异常室 | 损坏设备/奇怪痕迹 | 5% | T2-T3 |

## 3.2 难度等级

| Tier | 出现位置 | 敌人 | 危险地形 | 信息复杂度 |
|------|----------|------|----------|------------|
| **T1** | Run 第 1-2 房间 | 0-1 弱敌 | 无 | 低（2-3 模态混用） |
| **T2** | Run 第 3-5 房间 | 1-2 中敌 | 少量 | 中（多模态冲突） |
| **T3** | Run 第 5-7 房间 | 2-3 强敌或 1 精英 | 多处 | 高（伏击/诱饵/记忆触发） |

# 4. MVP 房间模板（10 个）

> 这是 MVP 必备的 10 个房间。M1 只用 R01。M5-M8 逐步上线其他。

## 4.1 R01-Lab — 初心房（M1 验收剧本专用）

**类别**：Explore + 弱遭遇  
**难度**：T1  
**尺寸**：20×20  
**对应**：[Sensory_Design §7 60 秒首房脚本](Sensory_Design.md#7-m1-验收剧本60-秒首房脚本)

```
            ┌─────────出口──────────┐
            │                       │
            │  [资源]    [威胁残影]  │  ← 顶部
            │   ●          ☠        │
            │                       │
            ├──墙──┐         ┌──墙──┤
            │      │ 分叉    │      │
            │      └─┐    ┌─┘      │
            │        │    │        │
            │        ●起点          │
            └────────────────────────┘
```

**节拍配置**（必出）：
- B-Fork × 1（起点正前方）
- B-Resource × 1（左路尽头）
- B-Smell-Src × 1（右路，威胁方向）
- B-Corner × 2（左右路转角）
- B-Enemy-Patrol × 1（出口走廊，弱敌）
- B-Exit × 1（顶部）

**程序变量**：
- 资源类型随机（Data / Biomass）
- 敌人血量随机 ±20%
- 威胁气味源位置在右路 3 个候选点抽取

**预想路径**：
- 主路径：起点→分叉→左路→资源→出口走廊→弱敌→出口（45-50 秒）
- 备路径：起点→分叉→右路→嗅觉警告→绕回左路→资源→出口（55-60 秒）

**节拍密度自检**：60 秒主路径走完 = 10 个事件（分叉/资源可见/拾取/绕路/嗅觉触发/敌人出现/战斗接触/胜负/继续/出口）✓

# 5. 程序拼接规则

## 5.1 Run 拼接算法（M8）

```
1. 抽取核心零件（Run 开始）
2. 按 Run 长度生成节点链：[T1, T1, T2, T2, T2, T3, T3]（典型 7 房间）
3. 每个节点按 Tier 从模板库抽取：
   - Tier 与房间难度匹配
   - 类别按 §3.1 频率分布抽取
   - 已抽过的模板进入冷却（同 Run 内不重复）
4. 连接：上一房间的"主出口"连接下一房间的入口
5. 验证：拼接后路径无死锁、节拍密度达标
```

## 5.2 模板变体

每个模板有 3-5 个变体，区分：

- 节拍元素位置抖动（±2m）
- 可选节拍元素的随机出现
- 敌人/资源种类的随机抽取
- 旋转 / 镜像（4 种方向）

**重玩性主要靠变体 + 拼接，不靠"完全程序化"**。

## 5.3 节点元数据（M8 星图）

```csharp
public sealed class RunNode
{
    public string RoomTemplateId;
    public int Tier;
    public NodeType Type;     // Normal / Elite / Shop / Event / Boss
    public string[] BranchTo; // 下一可选节点（M8 星图）
    public LootProfile Loot;
}
```

# 6. 节拍密度自检规范

**每个新房间模板上线前必须通过"路径自检"**：

## 6.1 主路径自检

1. 沿设计师预想的"主路径"以生物默认速度走一遍（模拟 5 m/s）
2. 标记每秒发生的事件
3. 累计"中/强/极强"事件数
4. **达标线**：单房间主路径 30 秒内至少 6 个事件

## 6.2 备路径自检

1. 沿替代路径（绕路 / 拾取 / 战斗）再走一遍
2. 同样标记节拍
3. **达标线**：备路径不能比主路径少 30%

## 6.3 失败处理

| 自检失败 | 应对 |
|---------|------|
| 主路径事件数 < 6 | 加节拍元素 |
| 连续 8 秒无事件 | 强制插入中等节拍 |
| 备路径远低于主路径 | 删除该备路径或加节拍 |
| 单房间 > 40 秒 | 拆成两个房间 |

- 房间尺寸 20×20m 对应屏幕约可见 75%（55° 等距相机，焦点生物居中）
- 长房间（30×8）需要相机沿走向略推 1.05x
- 任何房间不能有"屏幕外完全黑"超过 5 秒——玩家会失去方向感
